# 阶段1 MVP 开发需求文档

## 1. 文档目标

本文档用于指导“AI 驱动互动小说游戏”阶段1 MVP 的开发实现，目标是以最短路径交付一个可本地运行、可实际游玩的单机 Web Demo。

本文档严格按照原始作业文档中的“阶段一：MVP”范围编写，只覆盖以下能力：

- 完成世界设定包
- 实现初始化设定流程
- 实现选择题交互 + 流式输出
- 实现基础数值系统
- 实现至少 2 个章节的剧情推进
- 实现存档机制
- 落地基础上下文管理策略
- 稳定运行 30 轮以上

以下内容明确不属于阶段1 MVP：

- 50 轮稳定性兜底
- 章节进度指示器
- 完整游戏回放日志
- Token 消耗统计
- 图片、TTS、复杂动画、多结局、错误恢复等增强能力

## 2. 项目定位

### 2.1 产品定位

这是一个本地单机运行的 AI 互动小说 Web Demo。玩家扮演“现代工作牛马”，意外穿越到一个历史上不存在的古代王朝，从草民身份起步，凭借现代知识、古诗词积累、社会经验和临场抉择，在古代逐步改变命运，最终走向权势、财富与情感巅峰。

### 2.2 MVP 目标

MVP 不追求“大而全”，只验证三件事：

1. 玩家是否能顺畅完成“初始化 -> 进入剧情 -> 持续做选择 -> 剧情推进 -> 存档/读档”的完整闭环。
2. AI 生成的剧情是否能围绕设定持续推进，并让数值变化真正影响内容。
3. 系统是否能在 30 轮以上保持基本稳定，不明显鬼打墙，不频繁出现结构错误。

### 2.3 运行形态

- 运行方式：本地单机 Demo
- 用户模式：单用户
- 网络要求：仅后端调用模型接口时需要联网
- 不考虑：注册登录、多端同步、多人共享存档、正式部署运维

## 3. 技术约束

### 3.1 技术栈

- 前端：React + TypeScript + Tailwind CSS
- 后端：Node.js + TypeScript
- 包管理器：pnpm
- 架构模式：前后端分离
- 模型接入：OpenAI Compatible SDK

### 3.2 模型接入原则

后端只依赖统一的 OpenAI Compatible 接口，不绑定单一供应商。

必须支持通过配置传入：

- `baseURL`
- `apiKey`
- `model`

兼容目标包括但不限于：

- 豆包兼容网关
- Qwen 兼容网关
- 其他 OpenAI Compatible 服务

### 3.3 开发原则

- 优先实现最短可运行链路
- 尽量减少 Agent 数量，阶段1 不追求多 Agent 架构复杂度
- 规则计算和状态维护尽量本地结构化处理，不把核心状态一致性交给模型“自由发挥”
- 先保证可玩与稳定，再考虑美化和扩展

## 4. 题材与世界设定包要求

阶段1 必须先产出一份可供程序消费的世界设定包。该设定包既是创作基础，也是 Prompt 的输入依据。

### 4.1 核心题材定义

- 主角身份：现代职场普通人/打工人
- 开局处境：穿越到架空古代王朝，以草民身份开局
- 核心优势：现代常识、管理思维、商业理解、古诗词储备、心理博弈能力
- 成长路径：生存 -> 立足 -> 出名 -> 结交权贵 -> 进入权力核心
- 情感方向：可与关键女性角色建立关系，包括“白富美”类型角色与“女帝”类型角色
- 叙事风格：爽感明确、节奏快、每几轮要有明显推进

### 4.2 世界设定包最低内容

阶段1 必须完成以下内容并形成文档或结构化配置：

- 王朝名称、时代氛围、社会结构
- 主城/村镇/官府/书院/商会等核心地点
- 至少 4 名关键 NPC 档案
- 主角可选的 2 至 3 个初始身份或天赋方向
- 基础数值定义与变化规则
- 至少 2 章的大纲
- 明确的创作护栏

### 4.3 推荐 NPC 结构

阶段1 建议至少包含以下角色类型：

- 势利但可利用的地方豪强
- 能提供资源或任务的贵人角色
- 与主角存在情感线的女性角色
- 后续通向权力核心的关键人物

### 4.4 两章内容建议

为保证 MVP 快速落地，建议两章结构如下：

#### 第一章：草民立足

- 主角穿越落地，解决温饱和身份危机
- 利用现代知识或诗词能力获得第一次出头机会
- 结识第一个重要 NPC
- 获得初始资源、名望或人情

#### 第二章：初露锋芒

- 主角开始进入更高层级圈子
- 面临一次公开展示能力或站队选择
- 至少出现一次关系变化或资源置换
- 为后续“进入权力核心”埋下钩子

### 4.5 创作护栏

为减少模型跑偏，阶段1 必须约束以下边界：

- 不出现现代科技直接碾压式造物
- 不出现超自然修仙、系统外挂、无限资源等设定
- 不允许主角无代价瞬间封神
- 重要 NPC 的态度变化必须有因果
- 每轮选择都应推进局面，避免纯装饰选项

## 5. 阶段1 功能范围

### 5.1 初始化设定流程

玩家首次开始游戏时需要完成初始化。

必需输入或选择项：

- 角色名
- 初始背景
- 初始天赋方向
- 开局资源偏向

建议的初始背景选项：

- 落魄读书人
- 小商贩学徒
- 流民出身的机灵人

建议的天赋方向：

- 诗词才情
- 商业头脑
- 察言观色

建议的资源偏向：

- 一点碎银
- 一封人情旧信
- 一件可变卖旧物

验收要求：

- 初始化流程可在前端完成
- 所有选项会写入游戏状态
- 初始化完成后可进入首个剧情场景

### 5.2 章节游玩主循环

阶段1 的核心循环如下：

1. 后端根据当前状态构造 Prompt
2. 模型流式返回叙事文本
3. 本轮结束时返回 2 至 4 个可选项
4. 玩家点击选项
5. 后端基于选项更新状态并生成下一轮

每一轮必须产出：

- 一段叙事文本
- 2 至 4 个选项
- 可选的状态变化结果
- 可选的事件标签

验收要求：

- 玩家只通过点击选项推进剧情
- 选项数量不低于 2 个
- 叙事文本以流式方式展示
- 连续游玩至少 10 轮时无明显结构错误

### 5.3 基础数值系统

阶段1 只做轻量叙事数值，不做复杂战斗系统。

推荐基础数值如下：

- `reputation`：名望
- `wealth`：钱财
- `favor`：关键 NPC 好感映射
- `statusTags`：状态标签
- `inventory`：关键道具列表

其中：

- `favor` 至少支持 2 名关键 NPC
- `statusTags` 用于记录如“欠人情”“被怀疑”“小有名气”
- `inventory` 只保存对剧情有影响的关键物品

数值使用原则：

- 数值变化必须由玩家选择、事件结果或章节节点触发
- 数值变化必须能反向影响后续文案或选项
- 数值更新由本地代码执行，模型只负责提出候选变化

验收要求：

- 前端可看到关键数值的简化展示
- 至少有 3 类选择会引起数值变化
- 至少有 2 处剧情会读取数值并改变输出

### 5.4 存档机制

阶段1 采用本地单机最简单方案。

建议实现：

- 手动存档
- 手动读档
- 自动存档一个最近进度槽位

存档内容至少包含：

- 当前章节与场景标识
- 全量游戏状态
- 最近若干轮上下文摘要
- 最近一轮可供恢复的选项结果

存档介质建议：

- 后端本地 JSON 文件

验收要求：

- 玩家可在本地保存当前进度
- 重启前后可恢复继续游玩
- 读档后剧情上下文不明显断裂

### 5.5 基础上下文管理

阶段1 不做复杂记忆系统，但必须避免上下文无限膨胀。

建议采用“三段式上下文”：

- 固定设定层：世界设定、NPC 档案、章节目标、创作护栏
- 状态快照层：当前章节、当前场景、关键数值、关键关系、关键道具、已触发事件
- 近期历史层：最近 6 至 10 轮摘要

具体策略：

- 不把所有历史原文无限拼接进 Prompt
- 每轮结束后生成或更新一份结构化摘要
- 当轮 Prompt 优先使用状态快照与近期摘要，而非完整对话

验收要求：

- 连续游玩 30 轮时上下文长度可控
- 不因历史堆积导致明显性能恶化
- 剧情仍能记住关键已发生事件

## 6. 明确排除项

以下内容虽然在总作业中重要，但阶段1 不实现：

- 章节进度指示器
- 完整回放日志导出
- Token 统计
- 自由文本输入剧情
- 图片生成和 TTS
- 多结局复杂分支网
- 异常自动修复 Agent
- Thinking 或 Tool 可视化

排除原因：

- 这些能力不影响阶段1 对“核心游戏循环是否成立”的验证
- 会显著增加前后端复杂度，拖慢 MVP 首版落地

## 7. 系统方案

### 7.1 总体架构

采用前后端分离的单服务开发模式：

- 前端负责展示叙事、展示状态、接收玩家点击、触发存读档
- 后端负责状态维护、Prompt 构造、模型调用、流式转发、存档落盘

### 7.2 模块划分

前端模块建议：

- `pages` 或 `routes`：开始页、游戏页
- `components`：叙事面板、选项卡片、状态栏、存档操作区
- `stores`：当前游戏态与流式响应态
- `services`：调用后端 API

后端模块建议：

- `config`：模型配置与环境变量
- `game`：游戏状态、初始化、回合推进
- `prompt`：System Prompt 与用户态 Prompt 组装
- `llm`：OpenAI Compatible 调用封装
- `save`：存档读写
- `schemas`：请求响应结构定义

### 7.3 最简 AI 调用策略

阶段1 为了最快落地，建议使用单主模型回合生成，而不是多 Agent 编排。

每回合由一个主叙事调用完成：

- 生成叙事文本
- 生成可选项
- 输出建议状态变化
- 输出事件标签

然后由后端本地逻辑做二次校验：

- 校验结构是否合法
- 限制选项数量
- 校验数值变化是否超出边界
- 写入新状态

这样做的原因：

- 工程复杂度最低
- 容易排查问题
- 足够满足阶段1 MVP

## 8. 数据结构建议

### 8.1 GameState

```ts
type GameState = {
  gameId: string;
  playerProfile: {
    name: string;
    background: string;
    talent: string;
    startingAsset: string;
  };
  progression: {
    chapterId: string;
    sceneId: string;
    turn: number;
  };
  stats: {
    reputation: number;
    wealth: number;
    favor: Record<string, number>;
    statusTags: string[];
    inventory: string[];
  };
  storyFlags: string[];
  recentSummaries: string[];
  lastChoices: Choice[];
};
```

### 8.2 Choice

```ts
type Choice = {
  id: string;
  label: string;
  intent: string;
};
```

### 8.3 TurnResult

```ts
type TurnResult = {
  narrative: string;
  choices: Choice[];
  suggestedStateChanges: {
    reputation?: number;
    wealth?: number;
    favor?: Record<string, number>;
    addTags?: string[];
    removeTags?: string[];
    addItems?: string[];
    removeItems?: string[];
    addFlags?: string[];
  };
  events: string[];
};
```

## 9. 接口建议

### 9.1 初始化游戏

- `POST /api/game/start`
- 输入：角色名、背景、天赋、开局资源
- 输出：初始 `GameState` 与第一轮剧情

### 9.2 推进一轮

- `POST /api/game/turn`
- 输入：`gameId`、`choiceId`
- 输出：流式叙事文本，结束时返回结构化选项与最新状态

### 9.3 获取当前状态

- `GET /api/game/:id`

### 9.4 手动存档

- `POST /api/save`

### 9.5 读档

- `POST /api/load`

## 10. Prompt 设计要求

### 10.1 System Prompt 必须覆盖的信息

- 世界设定摘要
- 当前章节目标
- 关键 NPC 人设
- 数值含义
- 输出格式要求
- 创作护栏
- 节奏要求

### 10.2 回合 Prompt 必须包含的信息

- 当前状态快照
- 最近剧情摘要
- 玩家本轮选择
- 本轮需要推进的目标

### 10.3 输出控制要求

模型输出必须满足：

- 文风统一
- 不跳出戏外解释
- 不直接替玩家做决定
- 本轮必须给出明确可点的选项
- 尽量在每 2 至 3 轮制造一次明显推进

### 10.4 结构化输出建议

阶段1 推荐采用“文本流 + 结尾结构块”的方式：

- 流式部分输出叙事文本
- 结束时输出一个可解析的 JSON 块

这样可以兼顾：

- 前端打字机式流式体验
- 后端对选项和状态变化的结构化解析

## 11. UI 要求

阶段1 UI 只要求清晰、稳定、可玩，不追求重设计。

必须有以下区域：

- 顶部标题区
- 叙事文本区
- 选项按钮区
- 当前状态区
- 存档操作区

体验要求：

- 文本流式更新时页面不抖动
- 玩家在模型回复完成前不能重复提交
- 选项按钮在等待期间禁用
- 错误时有可理解的提示

## 12. 开发步骤拆解

以下步骤按实际开发顺序编排，建议严格顺序推进。

### 第 1 步：确定世界设定包

产出物：

- 世界观摘要
- 2 章大纲
- 关键 NPC 档案
- 数值定义
- 创作护栏

完成标准：

- 能直接作为 Prompt 输入
- 两章内容足以支撑 30 轮左右游玩

### 第 2 步：搭建前后端项目骨架

产出物：

- React 前端项目
- Node.js 后端项目
- 基础目录结构
- 环境变量模板

完成标准：

- 前后端可分别启动
- 前端能请求后端健康检查接口

### 第 3 步：定义核心类型与状态模型

产出物：

- `GameState`
- `Choice`
- `TurnResult`
- 存档结构

完成标准：

- 前后端共享类型定义或保持严格一致
- 本地可构造一份假数据驱动页面

### 第 4 步：实现初始化流程

产出物：

- 初始化表单页
- 创建游戏接口
- 初始状态生成逻辑

完成标准：

- 玩家完成选择后可以进入游戏页
- 首轮剧情可成功生成

### 第 5 步：实现 LLM 调用与流式输出

产出物：

- OpenAI Compatible 调用封装
- 流式响应通路
- 前端流式文本展示

完成标准：

- 能看到逐步输出的叙事文本
- 结束后能渲染模型给出的选项

### 第 6 步：实现回合推进与状态更新

产出物：

- 玩家点击选项后的回合推进接口
- 状态变化校验逻辑
- 章节内剧情推进逻辑

完成标准：

- 一次选择能推动剧情进入下一轮
- 数值变化能正确写回状态

### 第 7 步：实现基础数值展示和剧情联动

产出物：

- 状态栏 UI
- 数值变化规则
- 数值影响剧情的 Prompt 拼装

完成标准：

- 玩家能看见自己当前关键状态
- 不同数值会影响至少部分选项与剧情反馈

### 第 8 步：实现存档与读档

产出物：

- 本地 JSON 存档
- 手动存档按钮
- 手动读档入口
- 自动保存最近槽位

完成标准：

- 保存后可恢复
- 读档后可继续推进剧情

### 第 9 步：实现基础上下文压缩

产出物：

- 最近若干轮摘要逻辑
- 状态快照拼装逻辑
- Prompt 长度控制策略

完成标准：

- 连续游玩 20 至 30 轮时模型仍保持基本一致

### 第 10 步：进行 MVP 联调与稳定性自测

产出物：

- 一条完整通关或长流程试玩记录
- 一轮至少 30 回合的自测结果
- 已知问题清单

完成标准：

- 无频繁格式崩溃
- 无明显连续重复场景
- 可稳定完成阶段1 演示

## 13. 阶段1 验收标准

满足以下条件即可认定阶段1 MVP 完成：

- 有可运行的本地 Web Demo
- 玩家可完成初始化并进入剧情
- 叙事支持流式输出
- 每轮提供至少 2 个可点击选项
- 已实现基础数值系统并可见
- 至少有 2 章内容可推进
- 已实现存档与读档
- 已实现基础上下文管理
- 连续游玩 30 轮以上仍基本可用

## 14. 后续阶段预留

虽然阶段1 不实现以下内容，但开发时建议预留扩展点：

- 日志记录中间件
- Token 用量采集中间件
- 章节进度显示字段
- 更多章节配置加载机制
- 错误恢复与重试机制

这样可以在阶段2 少返工。

## 15. 结论

阶段1 MVP 的核心不是“做出一个完整商业化游戏”，而是尽快做出一个可跑、可玩、可连续推进的 AI 互动小说最小闭环。

本项目最重要的执行策略是：

- 题材先收敛
- 状态先结构化
- Prompt 先稳定
- 交互先跑通
- 存档和上下文管理尽早接入

只要先把这条主链路做通，后续阶段2 的日志、Token 统计、50 轮稳定性和体验增强才有可持续优化的基础。
